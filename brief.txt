Konsep Aplikasi Chatting Realtime + Sinkronisasi SQLite

1. Tujuan
   - Membuat aplikasi chatting realtime antar user.
   - Setiap pesan atau data user disimpan di database SQLite lokal.
   - RabbitMQ digunakan sebagai event bus untuk broadcast instruksi Create/Update/Delete (C/U/D).
   - Database di semua client sinkron 100% berdasarkan event RabbitMQ.

2. Struktur Database (SQLite)
   - Tabel messages:
       id INTEGER PRIMARY KEY AUTOINCREMENT
       username TEXT
       text TEXT
       timestamp TEXT
   - Tabel users:
       id INTEGER PRIMARY KEY AUTOINCREMENT
       username TEXT UNIQUE
       joined_at TEXT

3. Mekanisme RabbitMQ
   - Queue: "chat" (durable, agar tetap ada walau server restart).
   - Pesan: JSON dengan field:
       {
         "action": "create" | "update" | "delete",
         "type": "message" | "user",
         "sender": "<username pengirim>",
         "text": "<isi pesan>" (jika type=message),
         "timestamp": "<waktu ISO>",
         "joined_at": "<waktu ISO>" (jika type=user)
       }
   - Semua event dikirim ke queue "chat".
   - Semua client menerima event yang sama.

4. Logika Consumer (penerima event)
   - Terima event dari RabbitMQ.
   - Cek apakah event.sender == USERNAME (user lokal).
       - Jika YA → abaikan (tidak perlu C/U/D).
       - Jika TIDAK → lakukan C/U/D ke database sesuai instruksi.
   - Contoh:
       - action=create, type=message → INSERT ke tabel messages.
       - action=create, type=user → INSERT OR IGNORE ke tabel users.
       - action=update, type=message → UPDATE row di tabel messages.
       - action=delete, type=message → DELETE row di tabel messages.

5. Logika Producer (pengirim event)
   - Saat user login → kirim event "create user".
   - Saat user kirim pesan → kirim event "create message".
   - Event selalu menyertakan field "sender" = USERNAME.
   - Pesan dikirim dengan properti persistent (delivery_mode=2).

6. Sinkronisasi Database
   - Karena semua event broadcast ke semua client, dan setiap client hanya mengeksekusi event dari user lain, maka:
       - Database tiap client berisi semua pesan dari user lain.
       - Tidak ada duplikasi pesan sendiri.
       - Tabel users berisi semua user yang pernah login.
   - Dengan durable queue + persistent message, event tidak hilang walau client offline sebentar.

7. Alur Kerja
   - User A login → kirim event user.
   - User B login → kirim event user.
   - User A kirim pesan → kirim event message.
   - RabbitMQ broadcast event ke semua client.
   - Consumer di tiap client cek sender:
       - Jika sender != USERNAME → simpan ke SQLite.
       - Jika sender == USERNAME → abaikan.
   - Hasil: database di semua client sinkron, berisi data user lain dan pesan yang diterima.

8. Keuntungan
   - Realtime chatting antar user.
   - Database lokal tiap client sinkron tanpa polling.
   - Tidak ada duplikasi data sendiri.
   - Event-driven, efisien, dan konsisten.
